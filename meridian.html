Outbound Pipeline

Every outbound email follows this pipeline. Claude generates, rules engines validate, Roger approves, Graph API sends. Nothing leaves without human approval.

Email Composer P2
Selects contacts ready for outreach based on cooldown, bounce status, exclusions, and touch history. Passes org context (pain signals, messaging angles, site count, region) to Claude Sonnet for personalized draft generation. 30 emails per batch.
Follow-up Engine P3
Timer-based drafting. When a follow-up comes due (OOO return, "not now" revisit, manual timer), generates a contextual draft with full outreach history as input. Skips contacts with pending activity.
Draft Queue P2
Full lifecycle tracking: pending, approved, rejected, blocked, sent, failed. Stores violations, auto-fixes, Roger's edits, token usage, and Graph message ID. REQ-019: sent timestamp is confirmed delivery time, never draft time.
Send Engine P2
Graph API send on approval. Daily cap (40), per-org cap (3), 500ms rate limiting. Domain verification on first send (REQ-015). Logs outreach event with confirmed timestamp.
Enforcement Layer

Every piece of outbound content passes through two rules engines that enforce 15 requirements mechanically. No LLM needed. Pure regex, lookups, and state checks. Fast, free, and deterministic. Rules stored as JSON config, editable without code changes.

Content Rules Engine P2
Blocks forbidden phrases (timeline claims, financial language). Flags AI tells and em dashes. Enforces identity rules (Roger = CEO, Jonathon spelling), scheduling model, and augment framing. 11 REQs.
Email Rules Engine P2
Auto-fixes links (GoLegara.com format, no protocol prefix). Checks phone routing, exclusion registry, Brevo variables. Enforces cooldown, bounce blocks, daily and per-org caps. 4 REQs.
Governance Config P2
All rules stored as structured JSON. Forbidden phrases, approved alternatives, severity levels, and auto-fix patterns. Roger can edit rules without touching code. Changes logged to audit trail.

How Headless Browsers Work
What is a headless browser?
A headless browser is a real web browser (in Meridian's case, Chromium, the engine behind Google Chrome) running without a visible window. It does everything a normal browser does: loads web pages, executes JavaScript, renders content, handles cookies and authentication. The difference is that instead of a human clicking around, a program controls it.

Think of it as a research assistant that can visit any website, read its content, fill out forms, click buttons, and report back what it found. It runs in the background, completely automated.
How Meridian uses headless browsers today
Bounce resolution: When an email bounces, Meridian needs to find the correct address. The headless browser visits the organization's website, navigates to their staff/leadership page, reads the page content, and looks for email addresses that match the contact's name. If it finds one, it learns the email format for that entire domain (e.g., "this org uses firstname.lastname@domain.com") and applies that pattern to fix future bounces at the same org, without having to check the website again.
Future headless browser capabilities
Competitor monitoring: Periodically visit competitor websites. Detect messaging changes, new service announcements, pricing updates, and leadership changes. Build competitive intelligence automatically.

Live site verification: Check Legara's own website for stale content, broken links, deprecated terms, and governance violations. Run after every deployment. The site stays clean without manual QA.

Market research: Scrape public data sources (HRSA UDS reports, state licensing databases, 990 filings) to enrich org profiles with current data. Detect new FQHCs entering the behavioral health space, leadership changes, grant awards, and expansion announcements.

LinkedIn intelligence: Monitor target contacts' LinkedIn activity. Identify recent posts, job changes, and company updates. Prepare engagement talking points before outreach. (Connection requests and messages stay manual for TOS compliance.)
Architecture Principles
Three Intelligence Tiers
Route every task to the cheapest tier that can handle it. Rules engine (regex, lookups, state checks) for mechanical enforcement. Claude Sonnet for generation and classification. Claude Opus reserved for strategic reasoning. Most work never touches an LLM.
Governance as Config
Every rule, constraint, and enforcement policy is stored as structured JSON. Editable without code changes, logged to an audit trail with before/after values. The system can explain why it blocked or flagged something by referencing the specific rule. When a new lesson is learned, it becomes a rule. When a rule proves wrong, it's updated. The platform evolves.
Human in the Loop
The system generates, validates, and prepares. Humans approve and decide. No email sends without Roger's approval. No deal created without manual action. No rule changed without logging. Automation handles volume. Judgment stays with people. When a future BD team member joins, they see a task queue, not an architecture.
Specificity Over Generality
Meridian serves one methodology, one market (FQHC behavioral health), and one pipeline shape. Every module is built for this specific use case. That's why it works better than any off-the-shelf CRM, agent platform, or automation tool. The specificity is the moat.
